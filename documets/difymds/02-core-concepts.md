# Core Concepts

Understanding Dify's building blocks is essential before creating your first application.

---

## App Types

Dify offers two primary application types and three legacy types, all powered by the same underlying workflow engine.

### Primary Types

| Type | Description | Key Capability |
|------|-------------|----------------|
| **Workflow** | Single-turn task applications | Batch execution; processes discrete inputs and returns outputs |
| **Chatflow** | Multi-turn conversation applications | Persistent memory; conversation variables survive across turns |

### Legacy Types

| Type | Notes |
|------|-------|
| **Chatbot** | Simpler chat interface built on the workflow engine |
| **Agent** | Autonomous tool-calling interface |
| **Text Generator** | Single-turn text generation |

> **Tip:** For new projects, prefer **Workflow** or **Chatflow** тАФ they offer the most flexibility and the full node ecosystem.

---

## Start Node Types

Every workflow begins with one of two mutually exclusive start configurations:

| Start Type | Triggered By | Publishable As |
|-----------|--------------|----------------|
| **User Input** | Direct user interaction or API call | Web app, MCP server, API, tool |
| **Trigger** | Schedule, external event, or webhook | Internal automation only |

> Only **User Input** workflows can be published as standalone web apps or MCP servers.

---

## Chatflow vs. Workflow

| Aspect | Workflow | Chatflow |
|--------|----------|----------|
| Turns | Single-turn | Multi-turn |
| Memory | Resets after each run | Persists within a session via conversation variables |
| Trigger | User Input or Trigger | Every conversation turn |
| Output node | Output (End) | Answer |
| Best for | Document processing, data pipelines, automation | Chatbots, assistants, customer support |

## Variables

Variables are how data flows between nodes.

### Variable Types

| Type | Scope | Description |
|------|-------|-------------|
| **Input Variables** | Workflow start | Custom variables provided by the user |
| **System Variables** | Runtime | Pre-defined metadata injected automatically |
| **Output Variables** | Per node | Generated by each node; available to downstream nodes |
| **Environment Variables** | Workspace | Secure storage for API keys and secrets |
| **Conversation Variables** | Chatflow session | Persist across turns; managed via Variable Assigner node |

### System Variables

**Workflows:**

| Variable | Description |
|----------|-------------|
| `sys.user_id` | ID of the calling user |
| `sys.app_id` | ID of the current application |
| `sys.workflow_id` | ID of the workflow definition |
| `sys.workflow_run_id` | ID of the current execution run |
| `sys.timestamp` | Execution start timestamp |

**Chatflows add:**

| Variable | Description |
|----------|-------------|
| `sys.conversation_id` | Unique ID for the current conversation session |
| `sys.dialogue_count` | Number of turns in the current conversation |

### Variable Referencing

Variables can be inserted into any node input through three methods:

| Method | How To |
|--------|--------|
| Dropdown selector | Click the variable picker in a node's input field |
| Slash command | Type `/` in a text field to open the variable menu |
| Jinja2 syntax | Use `{{ variable_name }}` directly in prompt or template text |

## Triggers

Triggers are a special type of Start node that runs workflows automatically.

### Trigger Types

| Type | Activation |
|------|-----------|
| **Schedule Trigger** | Runs at predetermined times (cron or visual picker) |
| **Plugin Trigger** | Fires when an external system event occurs (e.g., GitHub PR, Slack message) |
| **Webhook Trigger** | Fires when an HTTP request hits the generated endpoint |

### Trigger Limits

| Plan | Max Triggers per Workflow |
|------|--------------------------|
| Sandbox | 2 |
| Professional / Team | Unlimited |

### Cron Expression Reference

```
тФМтФА Minute    (0тАУ59)
тФВ тФМтФА Hour     (0тАУ23)
тФВ тФВ тФМтФА Day of Month (1тАУ31)
тФВ тФВ тФВ тФМтФА Month       (1тАУ12)
тФВ тФВ тФВ тФВ тФМтФА Day of Week (0тАУ6, 0=Sunday)
тФВ тФВ тФВ тФВ тФВ
* * * * *
```

| Symbol | Meaning | Example |
|--------|---------|---------|
| `*` | Every | `* * * * *` = every minute |
| `,` | Multiple values | `1,3,5` |
| `-` | Range | `9-17` |
| `/` | Interval | `*/15` = every 15 minutes |
| `L` | Last | Last day of month |
| `?` | Any value | Used when day-of-month/week irrelevant |

**Shortcuts:** `@hourly` ┬╖ `@daily` ┬╖ `@weekly` ┬╖ `@monthly` ┬╖ `@yearly`

> Schedule triggers update `sys.timestamp` at each execution start but produce no output variables.

## Execution Models

### Serial Execution

Nodes connected in a chain execute one after another. Each node waits for the previous to finish.

```
[User Input] тЖТ [LLM] тЖТ [Code] тЖТ [Output]
```

### Parallel Execution

When multiple nodes connect to the same source node, they all run simultaneously.

```
                 тФМтЖТ [LLM A] тФАтФР
[User Input] тЖТ тФАтФАтФд            тФЬтЖТ [Variable Aggregator] тЖТ [Output]
                 тФФтЖТ [LLM B] тФАтФШ
```

**Limits:**
- Maximum **10 parallel branches** per node
- Maximum **3 levels** of nested parallelism

### Variable Scope in Parallel Branches

| Context | Access |
|---------|--------|
| Sequential flow | Downstream nodes read outputs from any prior node |
| Inside parallel branch | Cannot access other branches' variables |
| After parallel merge | All parallel outputs become available |

## Dify DSL (Domain-Specific Language)

All Dify applications export to **YAML format** тАФ the Dify DSL.

```yaml
# Example DSL structure
app:
  name: My Workflow
  type: workflow
nodes:
  - id: start
    type: start
    ...
  - id: llm_1
    type: llm
    ...
edges:
  - source: start
    target: llm_1
```

**DSL enables:**
- Exporting and importing apps between Dify instances
- Version control via Git
- Sharing workflows with the community
- Programmatic app generation

**DSL includes:** app configuration, workflow settings, model parameters, knowledge base connections

**DSL excludes:** API keys, knowledge base content, usage logs
